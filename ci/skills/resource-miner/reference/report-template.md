# 最终报告模板（硬约束 · 与 repo-miner 报告可解析字段完全一致）

下游 `build_reports_index.py` 无 frontmatter、靠固定 markdown 解析报告入库（站点/RSS/公众号都读它）。
**以下结构是硬契约，破坏即解析失败、报告无法上站。** 资源类报告只换「取值语义」，不换「骨架与行键」。

## 解析器依赖的 5 个硬约束

1. **H1 标题**：首个 `# ` 行。解析器会剥掉尾缀「深度分析报告」当 title。→ 按 SKILL.md 标题规则写**创意标题，≤32 字，含 2+ 要素，不加「深度分析报告」尾缀**（解析器对有无尾缀都兼容，省略是为公众号点击率）。
2. **`> GitHub: <url>` 行**：紧跟 H1 下方。**必须有**，是去重主键 original_url 的来源。
3. **四个固定 `## ` 章节标题**（一字不差）：`## 一句话总结`、`## 值得关注的理由`、`## 项目展示`、`## 项目画像`。
   - 一句话总结：正文首个非空行 = summary
   - 值得关注的理由：前 3 个列表项 = highlights（用 `1.`/`-`/`*` 起头）
   - 项目展示：首个 `![](http...)` = 封面（无图则整节省略）
   - 项目画像：见下表
4. **项目画像表 = 可解析行键白名单**（行键**一字不改**，只换取值语义）：

   | 行键（**不可改**） | 解析用途 | 资源类取值语义 |
   |---|---|---|
   | `GitHub` | — | 仓库 URL |
   | `Star / Fork` | 解析 stars/forks（逗号数字，`/` 分隔）| 真实 star/fork 数 |
   | `代码行数` | 解析 language + 存 loc_raw | **内容规模**：`N 条目 / M 文件 / X 体积`（主体文件类型），如 `CSS 91 bytes（仓库 14GB，181 epub）` |
   | `项目年龄` | 解析 age_months（`N 个月` 或 `N 年`）| `N 个月（YYYY-MM ~ YYYY-MM）` |
   | `开发阶段` | 存 stage 原文 | **更新节奏**：`活跃更新 / 稳定运维 / 低维护 / 已停滞`（带说明）|
   | `贡献模式` | 存 contribution 原文 | `独立运营 / 小团队 / 社区驱动`（带 commits 占比）|
   | `热度定位` | 解析 heat（取「（/:」前首词）| `极小众 / 小众精品 / 中等热度 / 大众热门`（带说明）|
   | `质量评级` | 存 quality 原文 | **策展视角**：`策展[优秀/良好/一般] 更新[活跃/停滞] 自动化[高/低]` |
   | `许可证` | 存 license 原文 | 真实许可证或 `无 License（注明含义）` |

   - **行键必须齐全**（无数据填真实值或 `N/A`，不可缺行）；`代码行数`/`开发阶段`/`质量评级`/`热度定位`/`贡献模式` 是 md-only 字段（enrich 兜底不补），缺了站点就空。
5. **收尾深度**：正文 ≥4KB；命中 ≥2 个 `CORE_SECTIONS`（核心价值/风险/行动建议/知识入口/可复用/竞品/套利）；结尾不截断（最后一行有结束标点）。文件名不得以 `phase\d` 结尾。

## 子类型取值微调（项目画像「质量评级」与「代码行数」）

- **awesome**：`代码行数` = `N 条目 / M 个 md 文件 / 外链 K 条`；`质量评级` = `策展[优秀] 覆盖[广/专] 更新[活跃] 失效治理[有/无]`
- **learning**：`代码行数` = `N 章节 / M 文件 / 含代码示例[有/无]`；`质量评级` = `深度[入门/进阶] 体系[完整/碎片] 维护[活跃/停滞]`
- **atypical**：按子类（规范/模板库/文档站/媒体列表/服务）选最能体现规模的口径；`质量评级` 突出该类最关键质量轴

> 无论怎么微调，**行键不变**（始终是 `代码行数`/`质量评级`），只换括号内取值。
