# 把 SQL 倒过来写：10.8K star 的管道式语言 PRQL，纯 Rust 编译到 13 种方言

> GitHub: https://github.com/prql/prql

## 一句话总结

PRQL 是一门「管道式」的 SQL 替代语言：你按思考顺序从 `from` 开始一行行变换数据，由一个纯 Rust 编译器把它翻译成 13 种数据库方言的标准 SQL——不取代数据库、不绑定引擎、承诺永久开源。

## 值得关注的理由

1. **它从语法层根治 SQL 的「书写顺序 ≠ 执行顺序」**：SQL 让你先写 `SELECT` 却最后执行，子查询要倒着读；PRQL 把查询重组成线性管道（`from → filter → group → select`），书写顺序就是逻辑顺序，还补上了变量、函数、类型这些 SQL 缺失的抽象。
2. **工程范本价值高于「又一门新语言」**：六阶段多 IR 编译流水线、用 PRQL 自身书写标准库再按方言模块覆盖、整个编译器编进 WASM 在浏览器里跑 Playground、文档代码块全部编译进快照测试——每一个都是可直接迁移到自己项目的设计模式。
3. **作者背书与定位都很硬**：核心作者是 xarray 维护者、Rust 快照测试库 insta 的作者，带着量化数据工程一线对 SQL 的切身痛感立项；项目明确「绝不商业化、不做语义层」，是数据栈里一个纯粹的「编译前端层」。

## 项目展示

PRQL 最好的「一图说清」素材是官方在线 Playground：左边写 PRQL、右边实时（浏览器内 WASM 编译）显示生成的 SQL。

## 作者视角：为什么存在这个项目

### 创始人/作者背景

仓库 owner `prql` 是 GitHub 组织 PRQL，真正的核心作者是 **Maximilian Roos**（GitHub: [max-sixty](https://github.com/max-sixty)），在本仓库约 2,100 次提交、占比近一半。他的背景组合很关键：

- **数据生态知名人物**：[xarray](https://github.com/pydata/xarray)（Python 科学计算的带标签多维数组库）核心维护者；
- **Rust 工具作者**：写了快照测试库 [insta](https://github.com/mitsuhiko/insta) 的相关生态工具，Rust 社区广泛使用——这直接塑造了 PRQL「文档即测试」的工程纪律；
- **量化数据工程经历**（Sixty Capital），长期被 SQL 的不可组合性折磨。

技术栈横跨 Python 与 Rust，对 SQL 痛点有切身体会——这正是「SQL 替代语言」立项的可信度来源。第二核心是 [aljazerzen](https://github.com/aljazerzen)（Aljaž Mur Eržen），编译器 / resolver 的关键贡献者。

### 问题判断

作者看到的是 SQL 作为数据变换事实标准的**结构性缺陷**：书写顺序与执行顺序不一致、无法把查询片段抽象成函数复用、`WHERE`/`HAVING` 因聚合位置分裂、各数据库方言互不兼容。别人把这些当作「SQL 就这样」忍下来，他选择把 R 生态 `dplyr`/`tidyverse` 的管道范式搬过来。

时机判断也精准：现代数据栈（dbt / DuckDB / 云数仓）让「在 SQL 之上再加一层语言」第一次有了足够大的可集成宿主——所以 PRQL 选择**编译到 SQL 而非取代数据库**，踩的正是这个窗口。

### 解法哲学

明确的「少而正交的强原语」而非「功能完整」取向，几个**明确不做**的决定比 feature 列表更有信息量：

- **不做语义层 / semantic model**（这是与 Google Malloy 的根本分野，官方默认接受这一批评）；
- **永不商业化**（设计哲学五原则之一 Open，Apache-2.0、无 CLA、无企业版信号）；
- **简单优先于性能**：方言代码生成 `dialect.rs` 顶部注释直言「优先 generic 方言、生成的 SQL 可能啰嗦，但换来 much less complex translator」——明确用产物冗长换编译器简单；
- **正确性优先于速度**：为重写 resolver（编译器语义解析中端）主动放缓开发，宁愿停下来还架构债。

### 战略意图

这是一个**基础设施层语言**，不是终端产品，也无商业化意图（genuinely open）。战略落点是「寄生于别人的工具」：一个 Rust core（prqlc）通过 FFI / pyo3 / WASM 导出到 7 种语言绑定，并编译进浏览器跑 Playground，让 PRQL 长在既有数据生态里而非自建门户。一个值得单独点出的强信号：作者把自研 AI agent 框架接入了本仓库 CI（`.github/workflows/` 下有 8 个 `tend-*.yaml` 做 issue 三连 / PR 评审 / CI 自愈），用 AI 缓解「贡献高度集中在少数人」的可持续性困境。

## 核心价值提炼

### 创新之处

1. **DSL 自举的标准库 + 方言模块覆盖**（新颖度 4/5，实用性 5/5，可迁移性 5/5）：`semantic/std.prql` 用 PRQL 自身定义所有变换/算子，`sql/std.sql.prql` 用 s-string + `module {dialect}` 嵌套声明各方言的 SQL 模板，子模块按方言**覆盖**默认实现。新增一个方言差异 = 加几行 PRQL，无需碰 Rust。把「13 方言 × 上百函数」的组合爆炸从代码降维成数据。
2. **集合运算用语言原语自定义**（新颖度 5/5）：`intersect`/`remove`/`append` 不是内置关键字，而是用 `join` + `tuple_map`/`tuple_reduce` 在 PRQL 里推导出来的函数——证明少量正交原语足以表达 SQL 的集合操作。
3. **反向锚定（anchoring）切分算法**（新颖度 4/5，实用性 4/5）：`sql/pq/anchor.rs` 从输出列出发**反向遍历**管道，在遇到塞不进同一条 SELECT 的变换（如窗口函数后再 filter）时切成子查询/CTE，生成结构正确且尽量少嵌套的 SQL。这是编译器最难的部分之一。
4. **文档即快照测试**（新颖度 4/5，实用性 5/5，可迁移性 5/5）：从 README / book 抽取每个 PRQL 代码块、编译并与 insta 快照比对，用 ` ```prql error` 标签声明「该例应当报错」，还能把编译出的 SQL 回插文档。文档腐化在这里被工程化地杜绝。
5. **整个编译器编译到 WASM 驱动 Playground**（可迁移性 4/5）：JS 绑定只是 `#[wasm_bindgen]` 薄壳，浏览器里跑的是**真实编译器**而非服务端或简化版，做到零后端、所见即所得。

### 可复用的模式与技巧

- **多 IR 分层 + 每层可序列化**：编译器对外暴露 `prql_to_pl` / `pl_to_rq` / `rq_to_sql` 及各层 JSON 序列化，既是黑盒也是工具箱，外部工具可在任意 IR 层接入。适用于任何 transpiler / 查询引擎 / Lint 工具。
- **能力分级闸门**（`Dialect::support_level` 把方言分 Supported / Unsupported / Nascent）：对每个 target 显式声明成熟度，诚实暴露而非假装完整。
- **薄绑定 + 单核**：7 个语言绑定全是 FFI / pyo3 / wasm 薄壳，包同一个 Rust core——一处实现、多语言导出。
- **逃生舱设计哲学**：s-string 嵌入原生 SQL、`internal` 嵌入编译器内置——抽象覆盖不到时给一个显式后门，避免「全有或全无」。

### 关键设计决策

最值得学的是**六阶段多 IR 编译流水线**（LR → PR → PL → RQ → PQ → `sqlparser::ast` → string，见 `prqlc/prqlc/ARCHITECTURE.md`）。从「管道语言」到「带 WHERE/HAVING/子查询结构的 SQL」语义鸿沟极大，单步翻译会爆炸；PRQL 用 PL（信息丰富）→ RQ（强类型精简、面向后端）→ PQ（已切分成可落单条 SELECT 的原子管道）逐层下降，每个边界都能 JSON 序列化、独立测试、被外部工具接入。Trade-off 是阶段多、维护成本高（resolver 正因此重写），换来的是每层职责单一。「多 IR + 每层可序列化」是任何编译器/转译器的范本。

## 竞品格局与定位

### 竞品对比矩阵

| 维度 | PRQL | Malloy（Google） | dbt | KQL / Kusto（微软） |
|------|------|------|------|------|
| 本质 | 管道式语言，编译到 SQL | 语义建模 + 探索语言 | SQL 之上的工程编排 | 管道式分析查询语言 |
| 引擎绑定 | 中立（13 方言） | BigQuery/Postgres/DuckDB 为主 | 跨库（仍写 SQL） | 锁定 Azure Data Explorer |
| 语义层 | 明确不做 | 内建（核心卖点） | 无 | 部分 |
| 开放性 | 永久开源、绝不商业化 | 开源、实验性 | 开源 + 商业公司 | 闭源、厂商产品 |

### 差异化护城河

「管道范式 + 编译到全方言 SQL + 引擎中立 + 可嵌入」四合一，目前没有直接对手同时满足这四点；DSL 自举标准库使方言扩展成本极低；文档即测试 + AI CI 形成的工程纪律也是软性壁垒。

### 竞争风险

- **缺语义层**使其难进 BI / 建模场景（Malloy 占住了更高维度的位置）；
- **长尾 SQL 表达力未补齐**：递归查询（`WITH RECURSIVE`）虽已落地但仍标实验性、假定 Postgres 语义（对应长期 open 的 [#407](https://github.com/PRQL/prql/issues/407) 应理解为「通用/健壮递归未完成」），「完全替代 SQL」的承诺尚未兑现；
- **贡献高度集中**：主作者一人约半数 commit，resolver 重写期开发放缓，是最大的可持续性风险；
- **用户惰性**：SQL 太根深蒂固，新语言冷启动难。

### 生态定位

数据栈的「编译前端层」，寄生于既有数据库 / 工具而非自立门户。最现实的胜利路径是成为工具链（IDE / Notebook / dbt 模型）里的一个可选查询语言——它与 dbt 实际是互补关系，完全可以作为 dbt 模型里的查询语言叠加使用。

## 套利机会分析

- **信息差**：项目本身已是大众热门（10.8K star），不存在「早鸟热度套利」空间；但**中文社区对 PRQL 的深度介绍极度稀缺**——价值在内容稀缺度，而非热度时机。「SQL 替代语言」又是长青技术话题。
- **技术借鉴**：六阶段多 IR + 每层可序列化、DSL 自举标准库 + 方言模块覆盖、文档即快照测试、编译器 WASM 化——这四个模式可直接迁移到任何编译器 / 转译器 / 多后端 SDK / 文档工程。
- **生态位**：填补了「引擎中立、可嵌入、纯转译」的空白——既不像 Malloy 那样绑 BI 建模，也不像 EdgeQL 那样绑自家数据库。
- **趋势判断**：稳步增长，近两月开发回暖；踩在「现代数据栈让 SQL 上层语言有宿主」的趋势上，相比 KQL（厂商锁定）、Malloy（绑建模）有「最纯粹、最开放」的后发差异化。

## 风险与不足

诚实地说，PRQL 离「日常替代 SQL」还有距离：官方自陈（README「Current Status」）目前较适合较简单查询、仍有 bug 与缺失特性；长尾 SQL 能力（健壮递归、部分窗口场景）未补齐；`unwrap()` 绝对量偏高（约 970 处，~452 在非测试代码，多集中在 resolver/lowering「不变量已建立」处，反映对内部不变量的强假设）；最现实的隐忧是贡献集中——一旦核心作者投入下降，项目节奏会明显受影响（团队也意识到这点，才用 AI agent 接管部分维护）。

## 行动建议

- **如果你要用它**：适合「想让团队（尤其非资深 SQL 用户）写出可读、可复用、跨方言查询」的场景，或作为 dbt 模型里的查询语言叠加使用；不要指望它替代需要语义层/建模的 BI 平台，也不要在依赖复杂递归/冷门方言特性的生产负载上 all-in。先用 Playground 试手感。
- **如果你要学它**：重点读 `prqlc/prqlc/ARCHITECTURE.md` 和 `src/lib.rs` 顶部的编译阶段图，然后看 `semantic/std.prql`（DSL 自举标准库）、`sql/std.sql.prql`（方言模块覆盖）、`sql/pq/anchor.rs`（反向锚定切分）、`web/book/tests`（文档即测试）这四处——它们是这个项目最高价值的浓缩。
- **如果你要 fork 它**：最有价值的改进方向是补齐语义层/建模能力（直面 Malloy 的差距）、健壮化递归与窗口的方言覆盖、以及降低 resolver 复杂度（团队自己也在重写）。
