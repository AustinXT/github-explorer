GitHub: https://github.com/masterking32/masterdnsvpn

一句话总结

一个荷兰独立开发者用 Go 写就的 DNS 隧道 VPN 工具，靠自研 ARQ 协议把单包头压到 5-9 字节（DNSTT 的 1/5、SlipStream 的 1/3），配合 8 种 DNS resolver 负载均衡策略，4.7 个月冲上 5,657 stars 并在伊朗 88 天断网期间保持可用，是 2026 年 GitHub 上 DNS 反审查领域最显眼的项目。

值得关注的理由

  1. 技术深度的确高：2,867 行自研 ARQ 协议 + 自研 base36 编码 + 零 DNS 库依赖（go.mod 仅 5 个三方包），工程成熟度不输 DNSTT、SlipStream 这些成名更早的项目。
  2. 极简协议头带来实打实收益：本地 10MB 文件下载 0.27s（DNSTT 2.49s / SlipStream 0.98s），单包头 5-9B 在 DNS label ≤63 字符的硬约束下把「能塞多少有效负载」做到了接近极限。
  3. 真实的反审查战例：README 显式提到「在伊朗 88 天断网期间，MasterDnsVPN 是少数仍能用的翻墙工具」——这是同期 DNSTT/SlipStream 都没能公开验证的能力，背靠 5,657 stars 的真实用户基础。

  • MasterDnsVPN（Go, 5.6K ⭐, 本仓）— DNS 隧道 VPN
  • MasterHttpRelayVPN（Python, 3.8K ⭐）— HTTP 中继 VPN，姊妹项目
  • WoW 私服生态：TrinityCore 魔改（C++）、WoWSimpleRegistration（PHP）、WoW-Server-Relay（JS）
  • DNS 工具：dns_finder（Python）、masterstream_uptime（监控）
  • 杂项：BMI 银行流水查看器（JS）、PHP DBC 编辑器等

近 1-2 年明显从「WoW 私服生态」转向「反审查通信基础设施」（HttpRelayVPN → DnsVPN → uptime 监控），转型方向明确。

问题判断

作者在 README 直白点明：DNSTT 和 SlipStream 这类「前辈」在严苛网络条件（高丢包、过滤 DNS、跨 ISP）下稳定性不足——伊朗 88 天断网期间它们陆续失效，作者认为问题是「协议头太大」+「resolver 模式太单一」，所以从头重写了一个「在所有能想到的最差情况下仍能交付数据」的版本。这不是营销话术——2,867 行自研 ARQ 和 2,158 行 8 模式负载均衡是实打实的工程回应。

解法哲学

  • 零 DNS 库依赖：连 DNS 报文解析都手写，理由是「敏感网络工具应最小化供应链攻击面」——这是个明确的安全哲学选择，不是工程偷懒。
  • 极简协议头：把 5 个 flag 位压缩到 1B（vpnproto/parser.go:25-34），头校验用 1 字节非密码学异或和（parser.go:185-192），任何字节都是为传输效率服务的。
  • 不做什么：没有 domain fronting、没有 obfs4 流量混淆、没有 forward secrecy 握手、没有 0-RTT。复杂度被严格限制在「协议头 + ARQ + 负载均衡」三条主轴，其他维度一概不堆。

战略意图

FUNDING.yml 只挂了 GitHub Sponsors + 三链加密货币收款（TON/EVM/TRC20），README 顶部波斯语版本自称「商业项目可联系」——但无价格、无套餐、无 SaaS。Telegram 频道 t.me/masterdnsvpn 是分发主阵地。结合 1,741 followers / 关注 6 人 / 95% 单人提交的画像，作者战略就是：做能跑、能扩散、能打赏的「作品」型反审查工具——不靠订阅、不靠企业客户，靠技术影响力和社区反哺。

核心价值提炼

创新之处

按新颖度 × 实用性排序：

  1. 协议头 5-9B 的极致压缩（vpnproto/parser.go:25-34）：5 个 flag 位用 bitmask 表达、最小头 4B（控制包）/ 9B（数据流包），比 DNSTT 的 ~40B（KCP 24B + Noise 16B）低 78-88%，比 SlipStream 的 ~20B 低 55-71%。这是 DNS label ≤63 字符硬约束下的硬实力。
  2. 2,867 行自研 ARQ + NACK + 自适应 RTO（internal/arq/arq.go）：Jacobson/Karels 7/8+3/4 加权（arq.go:533-548）、数据重传退避 1.35x（控制包 1.25x、握手包 1.15x）、NACK gap recovery（arq.go:1919-2109）——这是 KCP 都缺的能力。
  3. 8 种 DNS resolver 负载均衡 + 多路径复制（internal/client/balancer.go:23-33）：从纯轮询到丢包+延迟混合评分，从单路径到 requiredCount > 1 时同包复制到 N 个 resolver——伊朗断网 88 天还能用的核心机制。
  4. 13 种 DNS RR 类型做隧道载体（dnsparser/policy.go:12-36）：A/AAAA/CNAME/MX/NS/PTR/SRV/SVCB/CAA/NAPTR/SOA/HTTPS/TLSA——支持 SVCB/HTTPS RR type 是高阶拟态（Mix of Imitation），让流量混入 EDNS 增强查询。
  5. 客户端 DNS 缓存 + FNV 32-shard（internal/dnscache/store.go:32-33）：服务端缓存 + 客户端缓存双层防护，patchResponseForQuery 在复用时替换 query ID 但保留 TTL。

可复用的模式与技巧

可直接迁移到其他项目的设计/代码：

  1. Adaptive RTO 算法（arq.go:533-548）— 任何需要 RTT 估计的可靠协议（IoT CoAP、自研 MQTT）都可直接套用。
  2. TCP-like FIN/RST 三段握手（arq.go:1357-1400, 2576）— 9 个 StreamState 完整映射 TCP 半关闭语义，比 QUIC 的 connection ID 简单 100 倍但足够用。
  3. 多 shard map + FNV hash 跟踪 in-flight 请求（balancer.go:99 的 16-shard + dnscache/store.go:32 的 32-shard）— 高频读写表的标配。
  4. 自研 base36 编码（lowerbase36.go）— 7-byte block → 11-char base36，零外部依赖 + 自定义大小写解码映射，DNS/URL 短链/二维码/邮件附件名通用。
  5. DNS label 长度二分搜索（transport.go:422-441 CalculateMaxEncodedQNameChars）— 把 domain 长度 + label 切片关系变成可查表常数。
  6. AES-GCM + sync.Pool cipher 复用（security/codec.go:37-57）— Go 加解密性能优化标准模式，作者用 buffer pool 减少 50%+ GC 压力。
  7. packet flag bitmask + 256-lookup table（vpnproto/parser.go:256 buildPacketFlags()）— O(1) 类型查询，新增 packet type 只需改一处表。
  8. 薄入口 + 厚 internal — cmd/{server,client}/main.go 仅 200-264 行，纯 flag 解析 + config 加载。所有业务逻辑在 internal/，连 init() 函数都没有。

差异化护城河

  • 协议头 + Resolver 模式的双重优势——DNSTT 想抄协议头就得重写 ARQ（推翻 KCP 选择），SlipStream 想抄 resolver 多模式就得放弃 QUIC（推翻拥塞控制假设）。两个护城河互相加强。
  • 实战打磨的反向验证——同期项目里只有 MasterDnsVPN 有「伊朗 88 天断网仍可用」的自述战例，5,657 stars 的真实用户分布本身就是质量背书。
  • 工程化成熟度——CI 矩阵 20+ 平台、二进制 + Docker + 一键脚本三种分发、厚测试 + 631 行 bench、46 tag/37 release，单兵能做出这种工程度本身就是壁垒。

竞争风险

  • DNSTT 加 NACK + 多 resolver 是最大潜在威胁——如果作者 SagerNet 在 KCP 之上补这两个能力，MasterDnsVPN 的差异化优势会大幅缩水。
  • SlipStream 走 QUIC 路线 在「GFW 升级到能稳定识别 QUIC 指纹」之前都不是问题，但一旦发生 QUIC 集体失灵，MasterDnsVPN 未必能全身而退。
  • iOS 客户端缺失（issue #166） 是产品端最大风险——竞争对手都是 5+ 平台覆盖，单点缺位会被用户用脚投票。

生态定位

在整个 DNS 反审查工具生态中，MasterDnsVPN 填补的是 「工程化极简 + 多路径自适应 + 真实战例验证」 的空白——DNSTT 偏「极客工具」、SlipStream 偏「QUIC 实验场」、iodine 偏「经典教学」，MasterDnsVPN 是第一个把自己定位为「伊朗/俄罗斯用户能直接装的反审查 VPN」的工具。

套利机会分析

  • 信息差：5,657 stars 但 4.7 个月龄，在中文技术社区几乎无人讨论——公众号读者大概率是第一次见到这个项目。信息差套利窗口很大。
  • 技术借鉴：自研 ARQ + NACK + adaptive RTO 是国内 IoT / 弱网通信项目直接能复用的工程模板。Go 项目「薄入口+厚 internal + 零依赖」的组织方式也值得借鉴。
  • 生态位：填补了「DNSTT 太老、SlipStream 太 QUIC、iodine 不会有人用」中间地带的中端反审查工具空白。同作者 MasterHttpRelayVPN (3.8K⭐) 形成「HTTP 隧道 + DNS 隧道」姊妹组合。
  • 趋势判断：star 增长曲线是 4.7 月内「近 100 星集中一周内」的爆发型——热度仍在上升期，符合「近期爆款」标签。比 DNSTT 后发 4 年但已 star 反超，比 SlipStream 同期多 80%。

风险与不足

  1. bus factor 1.0 — 95% 提交来自 Amin.MasterkinG 一人，作者若停更（已观察到 5-6 月仅 5 commits），项目立刻进入「只修不增」模式。
  2. 抗审查宣传略大于实测 — README 称「Resilience: Very strong」（核心目标），但实际无 obfs 混淆层、无 forward secrecy、无 domain fronting。加密档 0/1 (NONE/XOR) 实质是明文/混淆；ChaCha20 无 MAC 可被比特翻转攻击；AES-GCM 共享 key 无 PFS——安全等级明显低于 DNSTT 的 Noise IK。UDP/53 端口发往自有域名的 DNS query 仍是已知指纹特征，GFW 升级到能统计识别加密熵高的 label 长度后可能失灵。
  3. 9x 性能数据可信度存疑 — 本地 10MB benchmark 是单方面数据，作者控制竞品版本/参数；没给高丢包（30%+）+ 高 RTT（200ms+）场景数据。SlipStream 上传比例（3.25/0.98=3.3x）反比 MasterDnsVPN（1.7/0.27=6.5x）更优，说明 SlipStream 在上传并发优化上可能更激进——「9x 快」不是均匀优势。
  4. iOS 客户端缺失 — issue #166 是头号用户痛点，5+ 平台里只缺 iOS 这块移动端核心，商业化路径走不通。
  5. 协议头 5-9B 优势在高丢包下会缩水 — 自研 ARQ 在 50%+ 丢包下需要 NACK 多轮往返；伊朗 8 个 resolver 在极端环境下能用的可能仅 1-2 个，多路径复制会放大单包延迟。88 天战例的鲁棒性边界未公开。

行动建议

  • 如果你要用它：优先于 DNSTT 和 SlipStream——5-9B 头开销在「能搞到自有域名 + 53/UDP 开放」的场景下确实领先；选择 BalancingHybridScore 或 BalancingLossThenLatency 模式，不要用 NONE/XOR 加密档（默认 AES-GCM），客户端用 Linux 桌面/服务器（iOS 等官方客户端）。域名前置要求 + 自有 NS 委托是必须的前置成本。
  • 如果你要学它：必读三个文件，按价值排序：
    1. internal/arq/arq.go（2,867 行）— 自研可靠传输层，Jacobson/Karels RTO + NACK + 流状态机的完整实现
    2. internal/client/balancer.go（2,158 行）— 8 种负载均衡策略 + 多 shard map + 粘性路由状态机
    3. internal/vpnproto/parser.go（321 行）— 协议头压缩 + bitmask flag + 1B 头校验的设计哲学
  • 如果你要 fork 它：三个方向最值得发力：
    1. 补 iOS 客户端（issue #166 头号缺口）— 用 NetworkExtension 框架 + Go 交叉编译，潜在回报最大
    2. 加 forward secrecy — 把当前固定 key 换成 Noise IK 握手，复杂度可控但安全等级追上 DNSTT
    3. 加 obfs 混淆层 — 在 ARQ 之上叠一层 packet length normalization，对抗 DPI 统计识别
  • 如果你是 GFW 对面的用户：这可能是 2026 年最好的 DNS 隧道 VPN 候选，但准备好自有域名（NS 委托 + A 记录）和53/UDP 出口开放的服务器。
