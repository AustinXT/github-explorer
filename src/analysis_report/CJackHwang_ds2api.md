# 逆向 DeepSeek 的免费 API，火后即封号归档

> GitHub: https://github.com/CJackHwang/ds2api

## 一句话总结

ds2api 是独立开发者 CJackHwang 用 4.5 个月做出的 Go 协议适配中间件：它访问 DeepSeek 的**网页聊天接口**（而非官方付费 API），用纯 Go 重写其反爬 proof-of-work、伪装浏览器流量，再把对话能力包装成 OpenAI / Claude 兼容的「免费 API」，支持流式与工具调用。技术上确实有巧思（4658 star、1504 fork）。但它本质是**违反 DeepSeek 服务条款的逆向代理**：封号风险已被实证、与上游反爬打「猫鼠游戏」70 个版本，最终于 2026-05 **归档停更**。本文把它当**技术案例 + 风险警示**来读——不建议依赖，正路是 new-api/one-api + 官方低价 key。

## 值得关注的理由（与必须看清的风险）

- **技术上的看点**：把「非标准的网页对话流」实时重组成「符合 OpenAI 规范的工具调用 + 流式响应」（核心 `toolstream`/`toolcall` 的 sieve 筛取），并用纯 Go 复刻对方的 PoW 校验——这是真正有工程难度的协议适配。
- **⚠️ 但这是 ToS 灰色 + 封号实证的方案**：Issue #508「被营销号报道后 DS 各种杀号，加代理也露头秒」、#506「正常使用即被封」——**风险不是理论，是已发生的事实**。
- **⚠️ 已归档、注定短命**：70 个 tag 挤在 4.5 个月，是与上游接口/反爬赛跑的记录；2026-05-10 归档（#91 宣布无限期停更）。**依赖它 = 依赖一个已宣告死亡、上游一变就失效、无人修的项目。**

## 作者视角：为什么存在这个项目

### 作者背景

**CJackHwang**（个人开发者，homepage 指向 linux.do 技术论坛）。工程能力强（Go 高质量、模块化清晰、测试齐全），但其代表作集中在「**逆向网页接口 → 包装成免费 API**」这一灰色赛道（另有 `AIstudioProxyAPI` 等）。值得肯定的是：他做了详尽免责声明，并在风险升级后主动停更归档——比许多同类项目负责。

### 问题判断

开发者想要「一套 OpenAI 兼容接口」来统一调用各模型；而 DeepSeek 的网页版能免费聊天、官方 API 要付费。作者看到的「机会」是：把网页版的对话能力**逆向**出来、包装成标准 API，让人不花钱就能像调 API 一样用 DeepSeek。**但这个「机会」从一开始就建立在违反对方 ToS、且无法控制的私有接口之上**——这决定了项目的灰色性质与脆弱命运。

### 解法哲学

- **选择逆向网页接口而非官方 API**：这是争议与脆弱的根源——上游不欢迎、随时可变。
- **选择纯 Go 实现 PoW + utls 伪装**：在传输层贴近浏览器特征以通过反爬（协议适配的硬骨头）。
- **选择把重心放在 tool-call 流式重组**：让网页聊天流能当标准 OpenAI 工具调用用（真正的技术难点）。
- **选择一键自部署（Docker/Vercel/Zeabur）**：传播路径，也是高 fork 比的来源。
- **选择 AGPL + 防御性措辞**：README 自称「技术探索/协议适配参考实现」，淡化「逆向冒充官方 API」的核心事实。

### 战略意图

这类项目的「意图」是用技术手段绕过付费墙。但它的生命周期由上游决定：DeepSeek 一改接口/反爬、或迫于压力封堵，项目即失效。**70 版本的猫鼠游戏 + 归档停更，就是这个意图的必然结局。** 它不是可持续产品，而是一次注定短命的技术博弈。

## 核心价值提炼

### 技术上可借鉴的（仅限学习，不含使用）

1. **流式协议适配（最有技术价值）**：`toolstream`/`toolcall` 的 sieve——在逐 chunk 到达的文本流里实时识别、切出跨 chunk 的工具调用片段，重组为结构化输出，并容错处理 JSON 修复 / schema 归一 / XML/DSML 多格式。这套「把非标准流适配成标准 API」的思路，对做 LLM 网关/适配层有参考价值。
2. **Go 中间件工程**：chi 路由 + utls（TLS 指纹）+ brotli 解压 + gjson/sjson 流式 JSON + tiktoken 计数——精简依赖、测试齐全的中间件范式。
3. **测试代替注释**：每个 internal 模块成对带 `_test.go`，跨 Go/Node 交叉验证流解析。

### 关键设计决策

- **协议适配层是技术重心**（fix 31.5% > feature 23.5%）：精力主要消耗在「跟上游变化保持兼容」，这正是这类项目维护成本高、稳定性差的结构性原因。
- **逆向而非聚合**：与正规 AI 网关（聚合自有合法 key）本质不同——后者健康可持续，前者灰色短命。

### 差异化与「护城河」

ds2api 在逆向赛道里热度较高、tool-call 适配做得较细——但这条「护城河」毫无意义：**整个逆向「免费 API」赛道都是快速生灭的红海**，ds2api 的归档就是注脚。同类项目层出不穷又快速消亡，连部分同行作者都在 README 里直接劝「官方贼便宜，建议走官方」。

### 生态定位

它是「逆向白嫖」现象的一个高热度切片，技术上值得当案例研读，实践上是反面教材。真正健康的生态位属于 new-api/one-api/LiteLLM + 各家官方 key。

## 套利机会分析（反向警示）

- **「信息差」在风险侧，不在收益侧**：读者极易被高 star + 「免费」诱导而**高估可用性、低估三重风险**（封号 / 上游一变即失效且已归档无人修 / 商业使用的法律合规）。star 越涨越说明营销号在放大，反而加速上游封堵。
- **算账其实不划算**：DeepSeek 官方 API 约 $0.28/百万输入、$0.42/百万输出，缓存命中与夜间错峰更低——**为省这点钱去冒封号/合规风险，得不偿失**。
- **技术借鉴 > 工具使用**：值得借鉴的是「流式协议适配」的工程思路，可迁移到合规的 LLM 网关；而不是部署这个已死的代理。

## 风险与不足（本项目的重点）

- **⚠️ 违反 DeepSeek ToS + 封号实证**：#508「火后 DS 大规模杀号、加代理也露头秒」、#506「正常使用即被封」——账号是被封的代价承担者。
- **⚠️ 已归档、上游一变即失效**：70 版本猫鼠游戏后停更，无人维护；DeepSeek 改接口/PoW 即彻底失效。
- **⚠️ 法律/合规灰色**：逆向冒充官方 API、绕过付费，商业使用风险尤甚；README 的「技术探索」措辞是规避争议的防御性框架，别被淡化。
- **稳定性天然差**：协议适配类对上游高度敏感，fix 占比远超 feature 即印证。
- **内容安全**：本文仅作技术机制评述与风险警示，**不提供任何部署/绕过检测/解反爬的操作步骤**。

## 行动建议

- **如果你想要「统一 OpenAI 兼容接口」**：✅ **正路是 new-api / one-api / LiteLLM + 各家官方 key**（聚合你合法持有的 key，合规、稳定、可商用）。DeepSeek 官方 API 本身极便宜，直接用官方即可。**不建议部署 ds2api**——它已归档、会封号、随时失效。
- **如果你只是想学技术**：可把 `internal/toolstream`/`internal/toolcall`（流式 tool-call sieve 重组 + 多格式容错）当作「把非标准流适配成标准 API」的工程案例研读，迁移到你**合规**的网关/适配层里。
- **如果你考虑 fork/依赖它**：⚠️ 不建议。已归档无人维护、上游一变即失效、封号与合规风险由你承担；AGPL 还要求你的网络服务也开源。
