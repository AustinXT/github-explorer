# ChinaTextbook 深度分析报告

> GitHub: https://github.com/TapXWorld/ChinaTextbook

## 一句话总结

用 GitHub 托管中国 K-12 全学段教科书 PDF 的公共资源库，为海外华人和教育研究者提供零门槛、免费、无水印的教材获取渠道。

## 值得关注的理由

1. **66,000+ Star 的资源型仓库**：罕见地以纯 PDF 资源而非代码获得大众关注，2025 年 5 月爆发式增长，展示了"资源数字化存档"的巨大需求
2. **Git 版本管理教育资源的独特范式**：将 Git 的版本控制能力应用于教材管理，教材年度更新可追溯——在竞品中独一无二
3. **安全工程背景驱动的工程化运营**：面对社区安全质疑，从"资源搬运"升级为"CI/CD 自动构建 + 开源工具链 + 独立仓库分离"的工程化方案

## 作者视角：为什么存在这个项目

### 创始人/作者背景

TapXWorld 是居住在东京的华人开发者，账号年龄 13 年（2013 年注册），具备安全工程/DevOps 背景（有 oswe/osep 安全认证工具仓库）。15 个公开仓库涵盖安全工具、区块链实验、快速部署环境等，但 ChinaTextbook 是其影响力最大的项目。主要贡献者 TESTPERSONAL（86% commits）疑为作者小号。

### 问题判断

作者作为海外华人，切身感受到获取中国教材的困难。项目初始于 2020 年（仅含数学教材），2022 年"人教版毒教材"舆论事件后，公众对教材内容透明度关注达到高峰，作者精准抓住窗口期，于 2023 年 1 月将项目从单一学科扩展为全学段全学科资源库。同时，作者观察到有人将国家平台免费资源加水印付费倒卖的现象，以此作为项目正当性叙事。

### 解法哲学

选择了"最笨但最可靠"的解法——直接将 PDF 文件托管在 GitHub 上：
- **选择做**：利用 GitHub 全球 CDN 和永久可用性，零成本分发；用 Git 版本管理追踪教材年度更新
- **选择不做**：不用 Git LFS（需付费存储配额），不依赖网盘（随时失效），不建工具（直接提供资源降低门槛）
- 宁可忍受 50MB 文件限制带来的分片复杂性，也要保持完全不依赖第三方服务的独立性

### 战略意图

项目经历了 Math（2020）→ ChinaTextbook（2023）→ ChinaTextbook + ChinaTextbook-tools（工具链分离）的演进。Telegram 社群、Star History 徽章、捐赠二维码都表明作者有意将其发展为可持续的社区驱动项目。在作者的 15 个仓库中，这是从个人 Side Project 成长为有媒体报道的公共资源项目的唯一案例。

## 核心价值提炼

### 创新之处

1. **用 Git 做教育资源版本管理**（新颖度 4/5，实用性 4/5，可迁移性 4/5）
   教材年度更新可追溯，如 2022 年"毒教材"替换为新版时，旧版仍可通过 Git 历史访问

2. **社会舆论窗口的精准把握**（新颖度 4/5，实用性 3/5，可迁移性 2/5）
   2023 年 1 月大规模扩展恰在"毒教材"事件之后，commit 中直接记录了"把'毒教材'移入旧版"的操作

3. **安全危机驱动的架构重构**（新颖度 3/5，实用性 4/5，可迁移性 4/5）
   面对 exe 安全审计请求 → 开源 Go 源码 → CI/CD 自动构建 → 分离工具链仓库 → 移除二进制文件

4. **二进制文件分片策略**（新颖度 3/5，实用性 3/5，可迁移性 3/5）
   自创 `_merge_folder` 命名约定 + 35MB 分片阈值 + 跨平台 Go 合并工具，绕过 GitHub 文件限制

### 可复用的模式与技巧

1. **"Git as CDN" 模式**：利用 GitHub 全球 CDN 托管静态资源，零成本高可用分发——适用于 <5GB 的公共资源库
2. **"分片+合并工具" 模式**：固定大小分片 + 命名约定 + 配套工具的三件套方案——适用于有文件限制的平台
3. **"学段/学科/版本/年级" 四级分类法**：多维度属性映射为目录层级——适用于教育资源库、多版本文档管理
4. **"安全审计驱动重构" 模式**：社区质疑 → 开源源码 → 自动化构建 → 分离关注点——适用于发布预编译二进制的开源项目

### 关键设计决策

| 决策 | Trade-off |
|------|-----------|
| GitHub 原生文件存储（非 LFS） | 增加分片复杂度，换来零成本和完全独立性 |
| 中文目录和文件名 | URL 编码后可读性差，但目录浏览体验极佳 |
| 工具链分离为独立仓库 | 用户需访问两个仓库，但解决了安全审计问题 |
| README 仅展示数学链接 | 其他学科需自行探索，但避免 README 膨胀 |

### 差异化护城河

ChinaTextbook 的护城河不是内容本身（同源于官方平台），而是**组织方式和分发基础设施的选择**。选择 GitHub 而非网盘，选择 Git 而非简单文件存储，选择结构化目录而非打包压缩——这些决策共同构建了"教育资源的版本控制系统"。66,000 star 的社区认知度形成了品牌效应。

### 竞争风险

- **版权执法**：如果国家中小学智慧教育平台发起 DMCA 或版权投诉，GitHub 可能应 takedown 请求移除仓库
- **官方平台改善**：如果官方平台开放海外访问并提供 PDF 下载，项目核心价值将被削弱
- SourceForge 镜像的存在说明社区已在做容灾准备

### 生态定位

在"已整理好的中国教材 PDF 资源库"细分领域事实上垄断。与 tchMaterial-parser 等下载工具形成互补而非竞争关系（README 主动推荐了 tchMaterial-parser）。

## 套利机会分析

- **信息差**: 项目已被充分发现（66,000 star），不存在信息差套利空间。但其资源本身具有长尾使用价值
- **技术借鉴**: "Git as CDN"模式、分片策略、安全危机应对流程可迁移到其他资源型项目
- **生态位**: 填补了"中国教材系统化数字存档"的空白，在版权灰色地带运营
- **趋势判断**: 仍在增长（月均 ~3,600 star），但仓库本身处于低维护状态（近 90 天 0 commit），增长来自社区传播而非内容更新

## 风险与不足

1. **版权风险**（最大）：教材 PDF 来自官方平台但未经授权分发，无 License 声明，随时可能被 DMCA takedown
2. **安全信任问题**：mergePDFs.exe 曾被 11/73 杀毒引擎标记（社区认为误报），虽已移除但影响残留
3. **可持续性堪忧**：依赖 1 人手动更新，2024-2025 年更新频率明显下降，仓库体积持续膨胀
4. **Commit 质量低**：近半数 commit 仅写"update"，贡献者身份混乱（test/Your Name/myname 等）
5. **文档不完整**：缺乏 CONTRIBUTING.md、Issue 模板、CHANGELOG，README 仅展示数学学科链接
6. **技术含量有限**：本质是 PDF 文件的 Git 托管，无源代码、无构建系统、无自动化采集

## 行动建议

- **如果你要用它**: 直接从 GitHub 浏览目录下载所需教材。超过 50MB 的 PDF 需要下载 `_merge_folder` 中的分片后用 [ChinaTextbook-tools](https://github.com/TapXWorld/ChinaTextbook-tools) 合并。建议本地保存一份，以防仓库被 takedown
- **如果你要学它**: 重点关注其"Git as CDN"分发策略、四级目录分类法、以及安全危机应对流程（Issue #35 的处理过程）
- **如果你要 fork 它**: 可改进方向包括：(1) 自动化采集流程（定期从官方平台同步新版教材）；(2) 完善 README 为全学科提供导航索引；(3) 添加搜索功能（GitHub Pages + 静态站点）；(4) 考虑 Git LFS 或外部存储缓解体积问题
