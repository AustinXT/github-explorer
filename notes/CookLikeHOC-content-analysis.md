## 动机与定位

- **要解决的问题**: 老乡鸡发布了一份约 20 万字的《菜品溯源报告》PDF，包含 226 个 SKU 的原料与制作流程，但 PDF 格式不可搜索、不可交互、不适合家庭厨师参考。项目将这些信息拆解为结构化的 Markdown 菜谱，并部署为可搜索的 VitePress 站点。
- **为什么现有方案不够**: HowToCook 等社区菜谱是「民间经验汇编」，配料量凭个人感觉，没有标准化数据源。CookLikeHOC 的差异在于数据来自真实商业餐饮企业的官方溯源报告，每道菜标注了精确克数、供应商名称（如成都圣恩生物、湖南鹏城远方食品）和中央厨房预制环节，这是社区创作菜谱无法提供的信息维度。
- **目标用户**: 对老乡鸡菜品感兴趣的家庭厨师，想在家复刻连锁餐厅味道的美食爱好者，以及关注餐饮供应链透明度的消费者。

## 作者视角

### 问题发现
核心维护者 SoilZhu（安徽某高校大二学生）在 Issue #26 中自述：受 HowToCook 项目启发，在老乡鸡发布《菜品溯源报告》后，出于「恶趣味/just for fun」心理，决定将 PDF 手动拆解为 Markdown 格式上传 GitHub。这是一个典型的「身边发现问题 + 已有模板可参考」的灵感触发路径——既有现成的社区菜谱仓库格式做参照（HowToCook），又有现成的高质量数据源（官方溯源报告），两者交叉催生了这个项目。

### 解法哲学
作者选择了「最小技术复杂度」的解法：不写爬虫、不做数据库、不搞 NLP 解析，而是人工逐篇阅读 PDF 并手动转写为 Markdown。这种「体力活 > 技术活」的策略在内容驱动型项目中反而高效——保证了数据准确性，避免了自动化解析可能引入的错误。VitePress 作为静态站点生成器，是这种纯文档项目的最优选择：零运维成本、部署简单、搜索体验好。

### 背景知识迁移
作者从两个领域带来了独特视角：
1. **程序员社区文化**：熟悉 GitHub 上的菜谱仓库生态（HowToCook 系列），理解 Markdown + 静态站点 = 低成本可持续维护的范式。
2. **消费者身份**：作为安徽籍学生，老乡鸡是身边最常见的连锁快餐品牌，这种「消费者解构消费品」的视角赋予了项目真实感和情感共鸣。

### 战略图景
从组织名「Gar-b-age」（垃圾团队的自嘲命名）和仅 3 个仓库的状态来看，CookLikeHOC 并非商业规划的产物，而是一次「玩票成名」的意外。Issue #81 中老乡鸡官方致谢并赠送 1000 元会员卡，进一步证实了这是个因出圈而被反向认可的社区项目。项目在作者更大的规划中没有商业化位置，更接近于「大学生的一次有趣的开源实践」。

共 199 篇菜谱 Markdown + 15 个分类目录，总计 3658 行 Markdown 内容。

### 关键设计决策

1. **决策**: 按烹饪方式（炒/蒸/卤/烤/煮...）而非食材分类
   - 问题: 199 道菜如何组织才能方便浏览？
   - 方案: 采用与餐厅菜单一致的「烹饪方式」分类，与老乡鸡溯源报告原始结构对齐
   - Trade-off: 想找「鸡肉相关菜品」需要跨多个分类搜索，但保留了与官方数据源的一致性，降低了录入时的分类决策成本
   - 可迁移性: 中 — 适用于任何源自既有分类体系的内容拆解项目

2. **决策**: 自动生成分类索引（generate-indexes.mjs）
   - 问题: 15 个分类的 README 需要列出所有菜谱链接，手动维护易遗漏
   - 方案: 编写 prebuild 脚本，在构建前扫描每个目录的 .md 文件，自动生成带拼音排序的索引 README
   - Trade-off: 引入了构建前步骤，但极大降低了内容维护成本；通过 `AUTO-GENERATED` 标记避免覆盖人工编辑的索引
   - 可迁移性: 高 — 任何 VitePress/文档站点项目都可直接复用此模式

3. **决策**: 动态导航与侧边栏生成（navSidebar.ts）
   - 问题: VitePress 默认需手动配置导航和侧边栏，新增菜谱需同步修改配置
   - 方案: 编写脚本在构建时扫描目录结构，自动生成 nav 和 sidebar 配置，按拼音排序
   - Trade-off: 失去了手动排序的灵活性（如将热门分类置顶），但实现了「添加 .md 文件即自动上线」的零配置发布体验
   - 可迁移性: 高 — 适用于所有基于目录结构组织的 VitePress 文档站点

4. **决策**: 保留供应链信息的诚实标注
   - 问题: 许多菜品使用中央厨房预制半成品或外部供应商调料包，配比未公开
   - 方案: 在配料中如实标注「老乡鸡中央厨房制作，详细成分与配比官方未公布，请依据个人口味适量调整」以及供应商名称（如「来自成都圣恩生物」）
   - Trade-off: 降低了家庭复刻的精确可行性，但保持了信息完整性和诚实度，反而成为独特的供应链透明度视角
   - 可迁移性: 低 — 高度依赖于特定数据源的信息结构

5. **决策**: Docker 多阶段构建部署
   - 问题: 社区用户希望快速自托管（Issue #129）
   - 方案: Node.js Alpine 构建 + Nginx Alpine 运行的两阶段 Docker 镜像，由社区贡献者（PR #141）实现
   - Trade-off: 增加了仓库复杂度，但满足了自托管需求；使用阿里云镜像源适配国内网络环境
   - 可迁移性: 高 — VitePress 项目的标准 Docker 部署模板

## 可复用模式

1. **目录即分类的 VitePress 文档组织**: 用文件系统目录结构直接映射内容分类，配合自动索引生成脚本消除手动维护成本 — 适用场景: 知识库、菜谱站、技术文档站等任何内容分类明确的静态站点

2. **诚实标注未知信息**: 对无法获取的精确数据（如配比、供应商配方），使用统一模板标注「官方未公布，请依据个人口味调整」，而非省略或编造 — 适用场景: 任何基于不完整数据源的内容整理项目

3. **Prebuild 钩子自动化**: 在 `npm run build` 前通过 `prebuild:indexes` 脚本自动生成派生文件，保持源文件与构建产物的一致性 — 适用场景: 需要索引、目录、统计等派生内容的文档项目

4. **社区贡献的基础设施补全**: 核心团队专注内容录入，Docker 支持（PR #141）、AI 配图版（PR #143）、侧边栏优化（PR #155）等工程改进由社区贡献者完成 — 适用场景: 内容驱动型项目的分工策略

## 竞品交叉分析

### vs HowToCook
- **CookLikeHOC 更好**: 数据权威性（企业官方溯源报告 vs 社区经验），精确克数标注（如「575g 鸡蛋液」「180g 大豆油」），供应链信息披露（标注中央厨房、外部供应商），聚焦单一品牌形成深度而非广度
- **HowToCook 更好**: 菜品覆盖面（社区持续贡献 vs 固定 199 道），实操友好性（家庭可复刻 vs 依赖商业调料包），社区活跃度（99k+ stars vs 约 16k stars），多维度教学（含厨房知识、食材处理技巧）
- **不同目标**: HowToCook 是「教程员学做饭」的通用教程，CookLikeHOC 是「解构商业快餐配方」的垂直参考。前者追求可操作性，后者追求信息完整性

### vs Cooklang / Mealie / Tandoor
- CookLikeHOC 是内容项目，后三者是工具/平台项目，不存在直接竞争
- Cooklang 的标记语言理论上可以用来结构化 CookLikeHOC 的菜谱数据，形成互补关系
- Mealie/Tandoor 的自托管菜谱管理功能可以导入 CookLikeHOC 的数据

### 综合竞争结论
- **差异化护城河**: 「企业官方溯源数据」是不可复制的数据源优势——除非其他连锁餐饮企业也发布类似报告，否则没有竞品可以提供同类信息
- **竞争风险**: 老乡鸡若更新菜单或撤回溯源报告，项目将失去数据源更新能力；项目声明「仓库主体部分于 2024 年完工」，后续更新动力有限
- **生态定位**: 在「中文菜谱 GitHub 仓库」赛道中占据独特的「企业配方解构」生态位，与 HowToCook 形成互补而非替代关系
