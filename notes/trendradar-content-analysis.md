## 动机与定位
- 要解决的问题: 信息过载时代，用户需要在微博、头条、知乎、B站等十余个中文平台间反复切换才能了解热点动态，且无法对信息进行个性化筛选和深度分析。核心痛点是"无效刷屏"——花大量时间浏览信息流，但真正关心的内容占比极低。
- 为什么现有方案不够: (1) newsnow/TopList 等聚合工具只提供"看"的能力，没有"推"和"筛"的能力；(2) RSSHub 虽覆盖广，但定位是 RSS 生成基础设施，用户需自行搭建阅读器和筛选规则，门槛高；(3) 没有现成工具将"多平台聚合+关键词筛选+AI 深度分析+多渠道推送"串成一条完整链路，尤其是面向中文生态。TrendRadar 的存在理由是将这条完整链路打包为一个"fork 即用"的产品。
- 目标用户: 中文互联网信息焦虑者——包括自媒体人（追热点）、投资者（盯行业信号）、企业品牌管理者（舆情监控）、技术爱好者（追踪技术动态）。实际核心用户群是有一定技术基础的中文开发者和信息工作者。

## 作者视角
### 问题发现
sansan0 的项目矩阵（bilibili 评论分析器、网易笔记备份工具、AI 代码上下文助手）全部围绕"信息采集-整理-分析"这一条主线。他显然是一位深度信息消费者，自己就是 TrendRadar 的第一个用户。从项目的细致配置选项（关键词正则、必须词/过滤词组合、时间段调度）来看，他是在长期实际使用中逐步打磨出这些功能的。问题发现的本质是"自用+开源"的经典路径。

### 解法哲学
作者的核心价值观是**极简部署 + 配置驱动**。与其让用户写代码定制，不如把所有变化点抽象为 YAML/TXT 配置项。这体现在：
1. **零代码门槛**: Fork 仓库 → 设置 Secrets → GitHub Actions 自动运行，全程不需要碰代码
2. **配置即产品**: config.yaml（26KB）+ timeline.yaml（27KB）+ frequency_words.txt 三个文件涵盖了所有定制需求
3. **渐进式复杂度**: 基础用户只需改 webhook URL；进阶用户可配置关键词规则、调度时间段；高级用户可启用 AI 筛选和 MCP 接口
4. **隐私优先**: 配置编辑器（index.html）是纯静态前端，数据停留在浏览器本地，不上传服务器

### 背景知识迁移
1. **信息聚合经验**: 从 bilibili 评论分析器和网易笔记备份工具积累的爬虫和数据处理经验，直接迁移到热榜数据采集
2. **中文互联网生态理解**: 对微博、头条、百度、B站等平台的热榜机制有深入认知，知道哪些平台的数据最有信息量
3. **用户运营感知**: 公众号"硅基茶水间"的运营经验让他理解目标用户的真实需求——不是技术极客，而是"想用但不想折腾"的普通人
4. **权重算法设计**: 将排名位置（60%）、出现频率（30%）、热度比（10%）融合为新闻权重的公式，体现了对热榜数据特征的经验性理解

### 战略图景
TrendRadar 在作者的项目矩阵中处于**流量入口+社区枢纽**的位置。它是 sansan0 所有项目中 star 数最多的，通过持续的版本迭代和社区互动建立了国内开源圈的个人品牌。从更新轨迹看，项目正从"热榜聚合推送工具"向"AI 驱动的个人信息雷达"演进——v5.0 引入 AI 分析，v6.0 引入 MCP 协议和 AI 智能筛选，标志着从"信息搬运"到"信息理解"的战略跃迁。MCP 接口的引入尤其值得注意，它将 TrendRadar 从一个独立工具转变为 AI Agent 生态中的一个可调用组件。

## 架构与设计决策

### 目录结构概览
项目采用清晰的六层架构，完全用 Python 实现：

```
trendradar/          # 核心包（~4500 行 Python）
├── __main__.py      # 入口+主协调器 NewsAnalyzer（~850行，偏重）
├── context.py       # 应用上下文，消除全局状态
├── crawler/         # 数据采集层（热榜 API + RSS）
├── core/            # 核心处理层（配置、分析、调度、频率词）
├── ai/              # AI 集成层（分析、筛选、翻译、LiteLLM 客户端）
├── notification/    # 通知分发层（9 个渠道的统一调度）
├── report/          # 报告生成层（HTML/Markdown 格式化）
├── storage/         # 存储层（SQLite 本地 + S3 远程，Mixin 复用）
└── utils/           # 工具函数

mcp_server/          # MCP 服务器包（~2500 行）
├── server.py        # FastMCP 2.0 入口
├── tools/           # 8 类 MCP 工具（数据查询、分析、搜索、配置管理等）
├── services/        # 数据/缓存/解析服务
└── utils/           # 校验和错误处理

config/              # 配置层（~70KB YAML/TXT，本身就是产品功能的一部分）
├── config.yaml      # 主配置文件（26KB，配置项极其丰富）
├── timeline.yaml    # 时间调度配置（27KB，含多种预设模板）
├── frequency_words.txt  # 关键词筛选规则
├── ai_filter/       # AI 筛选提示词
└── custom/          # 用户自定义目录
```

### 关键设计决策

1. **决策**: 寄生于 newsnow API 而非自建爬虫
   - 问题: 从十余个平台实时抓取热榜数据的工程成本极高，且面临反爬和 API 变更风险
   - 方案: 直接调用 newsnow 项目的公共 API（`https://newsnow.busiyi.world/api/s`），将数据采集外包给上游项目
   - Trade-off: 零采集成本、零维护负担，但完全依赖第三方服务的可用性和善意。一旦 newsnow 下线或限流，TrendRadar 的核心数据源即失效
   - 可迁移性: 低（依赖特定的上游项目，但"寄生式数据采集"的策略思路可复用）

2. **决策**: GitHub Actions 作为免费运行时
   - 问题: 让用户零成本运行定时任务，无需服务器
   - 方案: 利用 GitHub Actions 的 cron 调度能力，每小时执行一次爬取-分析-推送全流程。配合 7 天签到机制（Check In）控制资源消耗
   - Trade-off: 零成本运行换来了部署脆弱性（GitHub Pages/Actions 配置复杂度，见 #109/#513）、执行频率限制（最快每小时一次）、以及对 GitHub 免费额度的依赖
   - 可迁移性: 高（任何 cron 驱动的轻量级任务都可以采用此模式）

3. **决策**: 配置驱动的关键词筛选系统（frequency_words.txt DSL）
   - 问题: 用户需要灵活地定义"我关心什么"，但不应该需要写代码
   - 方案: 设计了一套文本 DSL——`+` 前缀表示必须词、`!` 前缀表示过滤词、`@数字` 控制显示数量、`/regex/` 支持正则、`=> 别名` 支持显示名称、空行分隔词组、`[GLOBAL_FILTER]` 区域定义全局过滤。这套语法简单但表达力强
   - Trade-off: 比 YAML 更直观易写，但语法不标准，需要文档教育成本；正则支持增加了出错可能
   - 可迁移性: 高（这套"轻量级文本 DSL 做配置"的思路适用于任何需要用户定义规则的场景）

4. **决策**: SQLite + S3 双后端存储，Mixin 复用逻辑
   - 问题: GitHub Actions 无状态（每次运行从零开始），需要持久化历史数据
   - 方案: 本地用 SQLite 存储当天数据，远程用 S3 兼容协议（支持 Cloudflare R2、阿里云 OSS 等）持久化。通过 `SQLiteStorageMixin` 将共用的数据库操作抽象为 Mixin，Local 和 Remote 后端分别继承
   - Trade-off: SQLite 简单可靠但不支持并发写入；Mixin 模式比组合模式（Composition）更紧耦合，但减少了代码重复
   - 可迁移性: 中（SQLite+S3 双后端的思路适用于任何"无服务器环境下需要持久化"的场景）

5. **决策**: LiteLLM 统一 AI 接口
   - 问题: 支持多种 AI 提供商（OpenAI、DeepSeek、Gemini、Claude、国内模型）而不为每个写适配器
   - 方案: 使用 LiteLLM 库统一 100+ 提供商的 API 调用，用户只需配置 `model: provider/model_name` 即可切换
   - Trade-off: 引入了较重的依赖（litellm 本身依赖链长），但极大降低了多模型适配的代码量
   - 可迁移性: 高（任何需要多 LLM 支持的项目都可以采用 LiteLLM）

6. **决策**: periods + day_plans + week_map 三级调度模型
   - 问题: 不同时段用户对信息的需求不同（上班看增量、下班看汇总、深夜不打扰）
   - 方案: 设计了灵活的调度系统：`periods` 定义时间段及其行为，`day_plans` 将时间段组合为日计划，`week_map` 将日计划映射到星期几。支持跨日时间段、预设模板和冲突策略
   - Trade-off: 配置灵活度极高（timeline.yaml 达 27KB），但学习曲线陡峭，普通用户可能被复杂度吓退
   - 可迁移性: 高（这套调度模型可直接搬到任何需要"时段差异化行为"的系统中）

7. **决策**: AI 智能筛选的两阶段架构（标签提取 + 批量分类）
   - 问题: 传统关键词筛选无法理解语义，用户难以用关键词覆盖所有感兴趣的话题
   - 方案: 阶段 A 用 AI 从用户的自然语言兴趣描述中提取结构化标签；阶段 B 用 AI 对新闻标题按标签进行批量分类。通过兴趣文件的 hash 判断是否需要重新提取标签
   - Trade-off: 用 AI 能力换来了更好的语义理解，但增加了 API 成本和延迟；两阶段设计让标签可缓存，减少了重复的 AI 调用
   - 可迁移性: 高（"先提取结构 → 再批量分类"的两阶段模式适用于任何 AI 驱动的内容分类场景）

8. **决策**: MCP 协议暴露 13 种工具
   - 问题: 让 AI Agent（如 Claude、Cherry Studio）能直接查询和操作 TrendRadar 的数据
   - 方案: 基于 FastMCP 2.0 实现 MCP 服务器，暴露数据查询、高级分析、搜索、配置管理、文章阅读等 8 类工具
   - Trade-off: 将 TrendRadar 从独立工具升级为 AI 生态组件，但增加了维护成本和依赖（fastmcp、websockets）
   - 可迁移性: 中（MCP 架构模式可借鉴，但依赖于 MCP 生态的发展）

## 创新点

1. **7 天签到制的自限机制**
   - 描述: GitHub Actions workflow 中内置了 7 天过期检查——超过 7 天没有手动 Check In，workflow 自动禁用。这不是技术限制，而是作者主动设计的资源节约机制，哲学含义是"如果 7 天都忘了签到，或许这些资讯对你来说并非刚需"
   - 新颖度: 4/5 | 实用性: 4/5 | 可迁移性: 5/5
   - 适用场景: 任何依赖公共 CI/CD 资源的开源项目，都可以用此模式避免僵尸 fork 浪费资源

2. **文本 DSL 驱动的关键词筛选引擎**
   - 描述: frequency_words.txt 用极简语法（+/!/@/[]/=>）实现了等效于 Elasticsearch 查询的筛选能力，包括必须匹配、否定过滤、正则、别名显示和数量限制。比 JSON/YAML 更易编辑
   - 新颖度: 3/5 | 实用性: 5/5 | 可迁移性: 4/5
   - 适用场景: 需要非技术用户定义内容筛选规则的任何信息聚合产品

3. **新闻权重三维计算公式**
   - 描述: 将排名位置（高排名得分更高）、跨平台出现频率（多平台热议 = 更重要）、热度比（高排名次数占比）三个维度融合为一个加权分数，且权重可配置
   - 新颖度: 2/5 | 实用性: 4/5 | 可迁移性: 4/5
   - 适用场景: 任何需要对多源信息进行融合排序的场景

4. **AI 筛选的标签提取-缓存-增量更新机制**
   - 描述: 对用户兴趣描述计算 hash，只有 hash 变化时才重新调 AI 提取标签；标签变化时使用 "update_tags" 提示词让 AI 给出差量更新方案（keep/add/remove），而非全量重建。这大幅减少了 AI API 调用成本
   - 新颖度: 4/5 | 实用性: 4/5 | 可迁移性: 4/5
   - 适用场景: 任何需要用 AI 维护结构化配置/标签的系统

5. **纯静态前端配置编辑器**
   - 描述: index.html 是一个不依赖后端的单文件 Web 应用，用户在浏览器中编辑配置，数据完全留在本地。通过 GitHub Pages 部署，实现了"隐私优先的 SaaS 体验"
   - 新颖度: 3/5 | 实用性: 4/5 | 可迁移性: 4/5
   - 适用场景: 任何需要提供配置 UI 但不想运行后端的开源项目

6. **多渠道推送的统一调度器 + 多账号配置**
   - 描述: NotificationDispatcher 用 `;` 分隔多账号、validate_paired_configs 验证配对一致性（如 Telegram 的 token 和 chat_id 数量必须相等）、limit_accounts 限制最大账号数。将 9 种渠道的差异性封装在 senders 层，上层统一调度
   - 新颖度: 2/5 | 实用性: 5/5 | 可迁移性: 4/5
   - 适用场景: 任何多渠道通知系统

## 可复用模式

1. **"Fork 即用"的产品化开源模式**: 利用 GitHub Actions + Secrets + Pages 构建完整的无服务器应用，用户通过 Fork + 配置 Secrets 即可获得个人化服务实例 -- 适用场景: 轻量级定时任务型工具（监控、爬虫、报告生成等）
2. **配置 DSL + 可视化编辑器**: 用简单文本 DSL 满足高级用户，用静态前端编辑器满足小白用户，两者编辑同一份配置 -- 适用场景: 需要同时服务技术用户和非技术用户的开源工具
3. **SQLite Mixin + 双后端模式**: 将 SQLite 操作抽象为 Mixin，让本地和远程后端共享数据逻辑，仅在连接管理上差异化 -- 适用场景: 需要同时支持本地和云端存储的应用
4. **AI 两阶段筛选（结构提取 + 批量分类）**: 先用 AI 从自然语言提取结构化标签，再用标签批量分类内容，标签层可缓存 -- 适用场景: AI 驱动的内容推荐/分类系统
5. **三级时间调度模型（periods → day_plans → week_map）**: 将时间段行为、日计划组合、星期映射三层解耦，支持复杂的差异化调度 -- 适用场景: 任何需要"不同时间做不同事情"的定时系统
6. **签到制自限机制**: 在免费基础设施上运行的开源项目，通过定期签到防止僵尸实例浪费资源 -- 适用场景: 依赖 GitHub Actions 等共享资源的开源项目

## 竞品交叉分析

### vs newsnow (ourongxing/newsnow, 18.9K stars)
- TrendRadar 更好: 主动推送（9 渠道）、关键词筛选、AI 深度分析、时间段调度、增量/汇总模式选择、MCP 接口
- newsnow 更好: 前端 UI 精美、实时更新、直接在线阅读、用户可自部署完整数据源、不依赖外部 API
- 不同目标: newsnow 是"看新闻的地方"，TrendRadar 是"新闻来找你"。TrendRadar 实际上是 newsnow 的下游消费者，两者构成互补关系而非直接竞争

### vs TopList (tophubs/TopList, 4.7K stars)
- TrendRadar 更好: 多渠道推送、AI 分析、个性化筛选、定时调度、持续维护中
- TopList 更好: Go 语言高性能、有预览网站 mo.fish、代码更轻量
- 不同目标: TopList 偏向"热榜聚合展示"的极简工具，TrendRadar 偏向"智能信息助手"的完整方案

### vs DailyHotApi (imsyy/DailyHotApi, 3.7K stars)
- TrendRadar 更好: 端到端方案（采集+分析+推送+AI），而非仅 API
- DailyHotApi 更好: 纯 API 设计更灵活、支持 Vercel 一键部署、支持 RSS 输出、可作为其他项目的数据层
- 不同目标: DailyHotApi 是中间件/API 层，TrendRadar 是面向终端用户的应用层

### vs RSSHub (DIYgod/RSSHub, 42.9K stars)
- TrendRadar 更好: 开箱即用的推送+AI 分析，不需要额外搭建阅读器；热榜排名和权重计算
- RSSHub 更好: 信息源覆盖极广（数千种）、社区生态成熟、可与任意 RSS 阅读器搭配
- 不同目标: RSSHub 是"万物皆可 RSS"的基础设施，TrendRadar 是面向"中文热点"的垂直应用。RSSHub 定位底层，TrendRadar 定位上层

### 综合竞争结论
- 差异化护城河: TrendRadar 的独特价值在于**端到端的信息处理链路**——从采集、筛选、分析到多渠道推送的一站式服务。竞品要么只做采集（newsnow/DailyHotApi），要么只做基础设施（RSSHub），没有竞品同时覆盖"AI 智能筛选 + 多渠道推送 + 时间段调度"这三个能力
- 竞争风险: (1) 对 newsnow API 的强依赖是最大的单点风险；(2) GitHub Actions 免费额度的政策变化可能影响核心部署模式；(3) AI 分析是差异化的关键，但随着 LLM 能力普及化，这一护城河可能被快速复制
- 生态定位: TrendRadar 处于**应用层**，位于 newsnow（数据源）和用户终端（微信/飞书/Telegram 等）之间。它的价值不在于某个单点功能，而在于将多个能力编排成一条顺畅的信息管道。未来通过 MCP 接口融入 AI Agent 生态，有望从"推送工具"升级为"信息感知组件"
