## 动机与定位

- **要解决的问题**: 中国义务教育阶段的教科书 PDF 资源虽然由国家中小学智慧教育平台免费提供，但获取渠道对普通人（尤其是海外华人）仍然受限。与此同时，有人将这些免费资源加上私人水印后在电商平台付费倒卖，形成了信息不对称的牟利空间。
- **为什么现有方案不够**: (1) 官方平台需要登录、有地域访问限制，海外访问速度慢或无法访问；(2) 网盘搬运不稳定，随时可能失效或需要付费会员下载；(3) tchMaterial-parser 等工具需要用户自行操作下载，有技术门槛；(4) 付费倒卖渠道存在水印和版权滥用问题。ChinaTextbook 提供了一个"零门槛、永久可用、Git 版本管理"的一站式方案。
- **目标用户**: (1) 海外华人家庭——希望孩子了解中国教育体系的家长；(2) 教育研究者——需要系统性获取各版本教材进行对比研究；(3) 国内偏远地区或网络受限的学生和教师；(4) 对"毒教材"事件等教育质量问题进行公民监督的公众。

## 作者视角

### 问题发现
TapXWorld 是居住在东京的华人开发者。项目初始于 2020 年 1 月（最初叫 "Math"，只包含数学教材），动机极其个人化——作为海外华人，他/她切身感受到获取中国教材的困难。2023 年 1 月是项目的转折点，从单一数学学科一举扩展到覆盖所有学段的全学科教材，这个时间节点恰好在 2022 年"人教版毒教材"舆论风暴之后，公众对教材内容透明度的关注达到了高峰。作者抓住了这个社会情绪窗口，将项目从个人需求升级为公共资源。

### 解法哲学
作者选择了"最笨但最可靠"的解法——直接将 PDF 文件托管在 GitHub 上。这个选择看似简单，实则体现了一种"基础设施思维"：不依赖任何第三方服务（网盘、CDN），不需要用户安装工具，利用 GitHub 的全球 CDN 和永久可用性，以及 Git 的版本管理能力来解决"教材年年更新"的版本追踪问题。作者宁可忍受 GitHub 50MB 文件限制带来的分片复杂性，也不选择 Git LFS 或外部存储，因为这样能最大限度降低用户获取资源的门槛。

### 背景知识迁移
作者具备安全工程/DevOps 背景（有 oswe/osep 安全认证工具仓库），这带来了几个独特 insight：
1. **CI/CD 思维**：曾为项目配置了 GitHub Actions release workflow，自动交叉编译 Go 语言的 PDF 合并工具到 Windows/Linux/macOS 多平台，这不是一般资源搬运者能想到的。
2. **安全意识**：当社区质疑仓库中 exe 文件的安全性时（Issue #35），作者迅速开源了 Go 源码，将 exe 从仓库中移除，改为通过 CI/CD 构建发布——这是典型的安全工程师思维。
3. **Git 工程能力**：用 Git 管理数 GB 的二进制文件并设计文件分片方案，需要对 Git 内部机制有深入理解。

### 战略图景
项目经历了清晰的演进路径：Math（2020）-> ChinaTextbook（2023）-> ChinaTextbook + ChinaTextbook-tools（工具链分离）。作者将"教材开源"定位为一种公共服务，而非技术项目。Telegram 社群的建设、Star History 徽章的展示、捐赠二维码的设置，都表明作者有意将其发展为一个可持续的社区驱动项目。在作者更大的规划中，这个项目是其 15 个公开仓库中影响力最大的一个，从一个开发者的侧项目成长为有媒体报道的公共资源项目。

## 架构与设计决策

### 目录结构概览
项目采用四层目录结构组织 2,375 个文件（其中 1,905 个完整 PDF 和 466 个分片文件）：

```
学段 / 学科 / 出版社版本 / 年级 / 文件名.pdf
```

顶层按学段分类：
- **小学**（1,053 文件）：10 个学科
- **高中**（598 文件）：18 个学科
- **初中**（548 文件）：18 个学科
- **初中（五四学制）**（70 文件）：8 个学科
- **小学（五四学制）**（66 文件）：6 个学科
- **大学**（19 文件）：高等数学、线性代数、离散数学、概率论
- **学数学最重要的刷习题在这里**（19 文件）：中考真题

覆盖约 176 个出版社版本，包含人教版、北师大版、苏教版、冀教版等全国所有主要教材版本。

### 关键设计决策

1. **决策**: GitHub 原生文件存储（而非 Git LFS 或外部链接）
   - 问题: 教材 PDF 单文件最大可达数百 MB，GitHub 限制单文件 100MB 上传
   - 方案: 将超过 50MB 的 PDF 拆分为每个 35MB 的分片（.pdf.1, .pdf.2, ...），放入 `_merge_folder` 子目录，并提供跨平台合并工具
   - Trade-off: 增加了用户使用复杂度（需要合并步骤），但换来了完全不依赖外部服务的独立性和永久可用性。仓库总体积约 3.95 GB（已追踪内容）+ 大量历史对象
   - 可迁移性: 中

2. **决策**: 中文目录和文件名
   - 问题: 用户是中文母语者，需要直观的资源定位
   - 方案: 完全使用中文命名（学段/学科/出版社/年级），文件名沿用官方教材全称（如"义务教育教科书·数学七年级上册.pdf"）
   - Trade-off: URL 编码后可读性差（%E5%B0%8F...），但用户浏览 GitHub 目录时体验极佳，且与官方教材名称完全一致，消除歧义
   - 可迁移性: 低

3. **决策**: 出版社版本命名规范——"版本名-出版社全称"
   - 问题: 中国教材存在同一学科多个出版社版本的情况，需要精确区分
   - 方案: 采用 `版本简称-出版社全称` 格式（如 "人教版-人民教育出版社"、"北师大版（主编：闫金铎）-北京师范大学出版社"），部分版本还标注主编姓名
   - Trade-off: 路径非常长，但信息完备，不会混淆
   - 可迁移性: 中

4. **决策**: 工具链分离（ChinaTextbook-tools 独立仓库）
   - 问题: 仓库中包含 exe 文件引发安全信任危机（Issue #35），且工具代码与资源文件混在一起不利于管理
   - 方案: 将 mergePDFs 工具移入独立仓库 ChinaTextbook-tools，通过 GitHub Actions 自动构建多平台二进制发布
   - Trade-off: 用户需要访问两个仓库，但解决了安全审计问题，也使主仓库更纯粹
   - 可迁移性: 高

5. **决策**: README 即全部文档，扁平化信息架构
   - 问题: 2,375 个文件的导航问题
   - 方案: README 仅以数学学科为样例提供直接链接，其他学科引导用户直接浏览目录。同时包含项目由来说明、文件合并教程、捐赠信息
   - Trade-off: README 只展示了数学科目的链接（94行），其余学科需要用户自行探索目录，但避免了 README 膨胀到不可维护
   - 可迁移性: 中

## 创新点

1. **用 Git 做教育资源版本管理**
   - 描述: 将 Git 版本控制应用于教育资源管理，使得教材年度更新可追溯（如 2022 年"毒教材"替换为新版时，旧版仍可通过 Git 历史访问）。2020 年初始版本仅含数学教材，2023 年扩展为全学科，138 次提交记录了完整演进过程。
   - 新颖度: 4/5 | 实用性: 4/5 | 可迁移性: 4/5

2. **二进制文件分片策略绕过 GitHub 限制**
   - 描述: 自创 `_merge_folder` 目录命名约定 + 35MB 分片阈值 + 跨平台 Go 合并工具的完整方案。123 个分片文件（占总文件的 19.6%）通过此方案管理。这种做法没有使用 Git LFS（需要额外的存储配额付费），而是纯粹利用 GitHub 免费层。
   - 新颖度: 3/5 | 实用性: 3/5 | 可迁移性: 3/5

3. **安全危机驱动的架构重构**
   - 描述: 面对社区对 exe 文件的安全审计请求（Issue #35），作者不是简单删除 exe，而是开源 Go 源码 -> 配置 CI/CD 自动构建 -> 分离工具链仓库 -> 从主仓库移除二进制文件。这一系列操作体现了从"资源搬运"到"工程化运营"的思维跃迁。
   - 新颖度: 3/5 | 实用性: 4/5 | 可迁移性: 4/5

4. **社会舆论窗口的精准把握**
   - 描述: 2023 年 1 月的大规模扩展恰好在"毒教材"事件之后，并在 commit message 中直接记录了"把'毒教材'移入旧版"的操作。项目将社会事件转化为项目发展动力，这是一种罕见的社区运营直觉。
   - 新颖度: 4/5 | 实用性: 3/5 | 可迁移性: 2/5

## 可复用模式

1. **"Git as CDN" 模式**: 利用 GitHub 的全球 CDN 基础设施托管静态资源文件，无需自建服务器或购买存储服务。配合 GitHub 的永久链接特性，实现零成本、高可用的资源分发 — 适用场景: 小型到中型（<5GB）的公共资源库、技术文档 PDF 合集、数据集分发

2. **"分片+合并工具"模式**: 当单文件超过平台限制时，采用固定大小分片 + 命名约定（.pdf.1, .pdf.2）+ 配套合并工具的三件套方案 — 适用场景: 任何需要在有文件大小限制的平台上托管大文件的场景

3. **"学段/学科/版本/年级"四级分类法**: 信息架构设计模式，将多维度属性映射为目录层级，用物理路径替代数据库索引 — 适用场景: 教育资源库、标准文档库、多版本多语言文档管理

4. **"安全审计驱动重构"模式**: 将社区安全质疑转化为架构改进动力——开源源码、自动化构建、分离关注点 — 适用场景: 任何发布预编译二进制的开源项目面对信任质疑时的标准应对流程

5. **"社会事件借势"模式**: 在公共事件引发关注时，快速扩展项目范围以承接社会需求（2022毒教材事件 -> 2023全学科扩展） — 适用场景: 公共利益型开源项目的社区增长策略

## 竞品交叉分析

### vs 国家中小学智慧教育平台（官方）
- 官方平台是权威来源，但需要登录、有地域限制、海外访问困难
- ChinaTextbook 以"镜像+归档"的方式弥补了官方平台的可访问性短板
- 官方平台内容更全更新更快，但 ChinaTextbook 的 Git 版本管理提供了官方不具备的历史追溯能力
- **竞争维度**: 可访问性 vs 权威性

### vs happycola233/tchMaterial-parser（4,748 Star）
- tchMaterial-parser 是从官方平台下载 PDF 的工具，定位是"钓鱼竿"
- ChinaTextbook 是已经下载好的资源集合，定位是"鱼"
- 两者形成互补而非竞争：ChinaTextbook 的 README 主动推荐了 tchMaterial-parser（"如果您位于内地......可以使用 tchMaterial-parser 重新下载"）
- tchMaterial-parser 的 Star 数更高（4,748），说明"工具型"项目在开发者社区更受欢迎；而 ChinaTextbook 的实际使用价值可能在非技术用户群中更高
- **竞争维度**: 工具 vs 资源

### vs hantang/smartedu-dl-go（22 Star）
- 功能类似 tchMaterial-parser 但使用 Go 语言实现，支持多平台
- 与 ChinaTextbook 同样是互补关系
- 从 Star 数看影响力远小于前两者
- **竞争维度**: 不构成直接竞争

### vs 网盘搬运（百度网盘/阿里云盘）
- 网盘搬运是 ChinaTextbook 最直接的竞品
- 网盘方案的劣势：链接失效、需要会员下载、无版本管理、有水印、难以系统化组织
- ChinaTextbook 的核心优势：永久链接、免费下载、Git 版本管理、结构化目录、无水印
- **竞争维度**: 持久性和规范性 vs 传播便利性

### 综合竞争结论
ChinaTextbook 的真正护城河不是内容本身（同源于官方平台），而是其**组织方式和分发基础设施的选择**。选择 GitHub 而非网盘，选择 Git 而非简单文件存储，选择结构化目录而非打包压缩——这些决策共同构建了一个"教育资源的版本控制系统"。在竞品格局中，它是唯一一个将"资源归档"和"版本管理"结合起来的方案，占据了独特的生态位。最大的风险在于版权争议（Issue #64）和 GitHub 平台政策变化。
